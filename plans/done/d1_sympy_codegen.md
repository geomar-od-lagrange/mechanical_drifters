# D1: Sympy -> numpy code generation

## Current state

The project has a working sympy derivation in
`src/drogued_drifters/lagrange_model.py`. It defines the Lagrangian from
scratch (T, V, drag forces, generalized forces), derives the EOM via the
Euler-Lagrange equations, extracts (M, F) with `linear_eq_to_matrix`, and
`lambdify`s them into numpy-callable functions.

However, the actual batch simulation code in `drifter.py::_rhs_batch()` does
**not** use `lagrange_model.py` at all. It has a completely hand-coded M and F,
written out element-by-element in numpy. The `lagrange_model` functions are
only used by the scalar `rhs()` method (via `_eval_M_F()`), which is the
slow single-particle path.

This creates three problems:

1. **Duplicated physics.** The batch path (`_rhs_batch`) and the scalar path
   (`rhs` -> `_eval_M_F` -> `lagrange_model`) encode the same EOM
   independently. Any change to the physics must be applied in two places.

2. **The batch code is unverified against the derivation.** The hand-coded M
   and F in `_rhs_batch` were transcribed manually from the sympy output. There
   is no automated check that they match. The `_rhs_batch` code also folds in
   the phi regularization (eps2 in M[3,3], damping in F[3]), which the sympy
   derivation does not know about.

3. **Startup cost.** `lagrange_model.py` re-derives everything from scratch on
   first call (~15s, as noted in the TODO). This is paid once per session due
   to `lru_cache`, but it means the sympy derivation currently serves no
   production purpose -- the batch path bypasses it entirely.

The goal is to make the sympy derivation the single source of truth by having
it **generate** the numpy code that `_rhs_batch` uses, then drop the
hand-coded version.


## Proposed approach

### Step 1: Generate static numpy code from sympy

Write a **code generation script** (`scripts/generate_eom.py`) that:

1. Runs the same Lagrangian derivation currently in `lagrange_model.py`.
2. Extracts the symbolic M matrix (4x4) and F vector (4x1).
3. Applies `sympy.cse()` (common subexpression elimination) to reduce
   redundant computation.
4. Uses `sympy.printing.pycode` (or `NumPyPrinter`) to render each matrix
   element / vector component as a numpy expression string.
5. Writes the result to a generated Python file,
   `src/drogued_drifters/_generated_eom.py`, containing two functions:
   - `compute_M(theta, phi, ...)` returning a dict or flat tuple of the
     nonzero M elements
   - `compute_F(theta, phi, xd, yd, thetad, phid, U_b, V_b, U_d, V_d, ...)`
     returning the 4 force components

The generated file will have a header comment like:
```
# AUTO-GENERATED by scripts/generate_eom.py -- DO NOT EDIT
# Re-generate with: pixi run python scripts/generate_eom.py
```

All arguments are scalars (or arrays -- numpy broadcasting handles both).

### Step 2: Vectorize via numpy broadcasting

The generated scalar expressions already work on arrays if the inputs are
arrays, because numpy arithmetic broadcasts element-wise. The key insight is
that sympy generates expressions like `m_d*l*sin(theta)*cos(phi)`, and if
`theta` and `phi` are `(N,)` arrays, numpy evaluates this as `(N,)`.

So the generated code is **automatically vectorized** -- no manual
broadcasting step needed. The `_rhs_batch` method just passes `(N,)` arrays
for all state variables and gets `(N,)` arrays back for each M element and F
component.

Assembly into the `(N, 4, 4)` M matrix and `(N, 4)` F vector stays in
`_rhs_batch`, since that's numpy array construction, not physics.

### Step 3: Handle phi regularization outside the generated code

The phi regularization (adding eps2 to sin^2(theta) in M[3,3], and the
dissipative torque in F[3]) is a **numerical technique**, not part of the
physical derivation. It should remain in `_rhs_batch`, applied after the
generated M and F are computed.

This keeps the sympy derivation clean (pure physics) and the regularization
explicit and easy to tune.

FB: Let's keep phi regularization outside for now. D2 is about reformulation (I'm leaning towards the stereographic projection). Then 
regularization would never be necessary if we don't expect (and we dont) the drogue to come out of the water.

### Step 4: Replace hand-coded `_rhs_batch`

Rewrite `_rhs_batch` to:
1. Call the generated `compute_M` and `compute_F`.
2. Assemble the (N,4,4) and (N,4) arrays.
3. Apply phi regularization.
4. Solve and return.

### Step 5: Remove `lagrange_model.py` runtime derivation

Once the generated code is the source of truth, `lagrange_model.py` with its
15-second startup derivation is no longer needed at runtime. The scalar `rhs()`
method can call the same generated functions (they work on scalars too).

Keep the derivation script as the generator, not as runtime code.

### Step 6: Verify

See verification strategy below.


## Key design decisions

### Generated .py file vs. lambdify at runtime

**Decision: generated .py file.**

- lambdify at runtime (current approach in `lagrange_model.py`) costs ~15s on
  first call and produces opaque lambda objects that can't be inspected or
  version-controlled.
- A generated .py file is inspectable, diffable, fast to import, and can be
  committed to the repo. The sympy derivation only runs when the physics
  changes.
- Tradeoff: the generated file must be kept in sync with the derivation script.
  Mitigated by a CI check (see below).

FB: Let's keep the lambdified version around for verification and for other explorative stuff, though. But afaik, this is the plan anyway.

### CSE or not

**Decision: use `sympy.cse()`.**

The M and F expressions share many subexpressions (sin(theta), cos(theta),
cos(phi), etc.). CSE will reduce redundant trig calls. The generated code will
have a block of intermediate variables followed by the final expressions.

### What about `sympy.physics.mechanics`?

The current derivation in `lagrange_model.py` uses `dynamicsymbols` but does
the Lagrangian mechanics manually (computing T, V, Q, then the Euler-Lagrange
equations by hand). The `sympy.physics.mechanics` module (Kane's method,
Lagrange's method) could automate this further.

FB: I like the explicity way we handle this based on dynamic symbols. This is immediately obvious to any physicist or engineer trainied in Lagrangian mechanics and generalized forces. The elegance of the .mechanics submod comes at a cognitive cost which is not worth it esp as we generate performant code anyway.

**Decision: stick with the manual Lagrangian approach.**

Reasons:
- The current derivation already works and is readable.
- `sympy.physics.mechanics.LagrangesMethod` would add another abstraction
  layer for marginal benefit -- the system has only 4 DOF.
- The drag forces are velocity-dependent and non-conservative, which requires
  careful handling in the mechanics module anyway.
- The manual approach matches the manuscript's notation more closely.


## Verification strategy

### V1: Numerical agreement with current code

Before removing anything, add a test that evaluates both the hand-coded
`_rhs_batch` and the new generated-code path at a grid of test points
(varying theta, phi, velocities, currents) and asserts they agree to
machine precision.

Test points should include:
- theta near 0 (drogue above buoy -- degenerate but should match)
- theta near pi (nominal operating point)
- theta = pi/2 (horizontal pole)
- Various phi values
- Nonzero velocities and currents
- Edge case: all zeros

### V2: Agreement between scalar and batch paths

Verify that `rhs()` (scalar, using generated code) and `_rhs_batch()` (batch
with N=1, using generated code) produce identical results. This is currently
not tested.

### V3: Sympy symbolic check

In the generation script, after computing M and F symbolically, verify:
- M is symmetric: `assert (M - M.T).simplify() == sp.zeros(4, 4)`
- M has the expected sparsity (M[0,1] = M[1,0] = 0, M[2,3] = M[3,2] = 0)
- Dimensions are correct

### V4: CI freshness check

Add a CI step (or test) that re-runs the generation script and asserts the
output matches the committed `_generated_eom.py`. This catches cases where
someone changes the physics in the derivation but forgets to regenerate.

## Risks and open questions

### R1: Expression size

The sympy-simplified expressions for the F vector (especially the drag terms)
involve `sqrt(...)` of velocity differences. After CSE, these should be
manageable, but the generated file could be large. Not a real problem -- it
just needs to be correct, not pretty.

### R2: The `|v - u|` drag term and sympy simplification

The quadratic drag `F = -k * |v - u| * (v - u)` produces `sqrt(...)` terms
that sympy may struggle to simplify. The current `lagrange_model.py` already
handles this (it uses `_mag` which is `sqrt(v.dot(v))`), so this is a known
quantity. The generated code will contain sqrt expressions -- that's fine for
numpy.

### R3: Do we commit the generated file?

**Yes.** This avoids requiring sympy as a runtime dependency and eliminates
the 15-second startup cost. The tradeoff is staleness risk, mitigated by the
CI freshness check (V4).

### R4: Should the derivation script also generate the drogue velocity expressions?

The hand-coded `_rhs_batch` computes `xd_d` and `yd_d` (drogue horizontal
velocity) manually. These are just `d/dt` of the drogue position, which sympy
already knows. The generation script could emit these too, but they're simple
enough (two lines) that it may not be worth the complexity. **Open question --
decide during implementation.**

FB: Keep simple for now. If we later find that it's worth adding xd_d etc. to the generated code, we'll do this then.

### R5: Interaction with D2 (phi regularization)

D2 recommends keeping the current phi regularization approach. The current plan
keeps regularization outside the generated code, which is the right separation.
D2 can modify the regularization in `_rhs_batch` without touching the
derivation. If D2 were to change the coordinate system (e.g. quaternions),
that would require changing the derivation too, but that is not recommended.

FB: I think D2 will go for stereographic. But don't worry for now and just assume we stick with the current approach. Nothin in D1 depends on the coord system for the drogue.
